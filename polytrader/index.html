Problem Statement

Prediction markets are rich in data, poor in tooling

Polymarket exposes every trade and every price tick over three different APIs, but the path from "I have an edge" to "I have an order on the book" is full of papercuts: corporate SSL proxies, broken page-size limits, V1→V2 migrations that silently kill order signing, and no batteries-included strategy/backtest framework.

!

Edges go un-harvested

Low-probability markets with mid-volume see frequent price spikes (whales eating books). With no infra to detect and counter-trade them, the mean-reversion edge sits unclaimed.

⚠

Crawler quirks

Trades API silently caps at 500 (not 10,000). Corporate Zscaler breaks SSL. Public Polygon RPCs time out. Every onboarding hits the same three walls.

↻

V2 migration shock

Apr-28 cutover wiped V1 open orders, changed the EIP-712 domain version, and broke every V1 client. No drop-in upgrade path; allowances must be re-approved for new exchanges.

⚙

Wallet plumbing

USDC.e vs pUSD. EOA vs proxy wallets. CTF approvals. Magic.Link signing quirks. None of this is the strategy — but all of it must be right before a single order can fill.

User Personas

Who this is built for

Polytrader is single-operator software — the same person crawls, backtests, and trades. The personas describe the three hats that operator wears.

Primary · Quant Operator

The strategy author

Wants to express "buy the dip on rare-event YES tokens" or "sell mean-reverting hazard rate spikes" as code, then have the system enforce gating: no live orders until backtest passes.

  • Define a Signal in < 100 lines
  • Run walk-forward backtest on 90 days of data
  • Promote from dry-run to live with one flag
Secondary · Researcher

The market analyst

Searches across thousands of markets, looks for low-probability events with stable lambda, and exports clean datasets to Jupyter. Does not necessarily trade — but feeds insights back to the operator.

  • Search-as-you-type across full market universe
  • Volatility/correlation/efficiency metrics per market
  • CSV/JSON export for external models
Tertiary · Wallet Custodian

The on-chain operator

The same person, with a different concern: making sure approvals are set, USDC.e is wrapped to pUSD, and that a runaway loop can't drain the wallet via 1,000 micro-orders.

  • One-time V2 allowance setup script
  • Hard caps: $5/trade, $20 total exposure, 10/hr
  • Dry-run by default, --live opt-in
User Journey & Pain Points

From cold clone to live order in four phases

Each phase has a gating point — failing the backtest threshold, for example, blocks the next phase. The system is opinionated about not letting a strategy near real money before it earns the right.

1

Crawl

Pull market metadata, price history, and trades from three Polymarket APIs into local SQLite. SSL patched, rate limits respected.

Pain: 500-row API cap, Zscaler proxy
2

Analyze

Search and visualize across thousands of markets via the web explorer. Find low-prob, mid-volume markets with stable lambda.

Pain: matplotlib doesn't scale to 10k markets
3

Backtest

Walk-forward simulator runs the strategy day-by-day. Output: win rate, Sharpe, profit factor, exit-reason breakdown.

Pain: gate at win>55% & Sharpe>1.0
4

Trade

TradingClient signs V2 EIP-712 orders via py-clob-client-v2. Dry-run by default; --live opt-in passes safety checks first.

Pain: $5/trade hard cap, never bypassable
User Stories

What the operator needs to accomplish

Every story maps to a CLI verb. The system optimizes for "express intent in one command" rather than for general-purpose flexibility.

As a quant operator
I want to crawl 90 days of trades for all $50K+ markets in a single command
so that I have a research dataset before I write a single line of strategy code.
As a strategy author
I want to subclass BaseStrategy and return a list of Signal objects
so that the engine handles entries, exits, and PnL accounting — I only write the alpha.
As a researcher
I want to search "election" across all markets and see live price curves
so that I can spot anomalous spikes without writing a notebook.
As a wallet custodian
I want every order capped at $5 and the daily count capped at 10
so that an off-by-one in the strategy can't drain the deposit address overnight.
As a V2-era trader
I want one script that approves pUSD + CTF on both V2 exchanges
so that I don't get "allowance: 0" errors three blocks before the news drops.

Performance

Crawler respects per-endpoint rate limits (50/10s markets, 100/10s prices). 90-day full crawl on 200 markets fits in a single session. SQLite indexed on condition_id + timestamp for sub-second strategy scans.

Scalability

Storage scales to 10k+ markets via the web explorer's pagination + server-side search. New strategies plug in by subclassing BaseStrategy — no changes to engine or executor.

Safety as architecture

The backtest gate is structural: execute reads the latest BacktestResult and refuses to send orders if metrics are below threshold. Safety isn't a habit — it's a code path.
